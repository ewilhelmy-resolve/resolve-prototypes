# RITA Commit Messages

Follow these conventions when creating commits for RITA.

## Prerequisites

### Step 1: Ask for Ticket Number

**Always ask the user for the ticket number** before proceeding. Accepted prefixes: `RG-`, `CLIEN-`, `JAR-`.

Example prompt: "What's the ticket number for this change? (e.g., RG-101)"

### Step 2: Check Current Branch

```bash
git branch --show-current
```

**If on `main` or `master`, create a feature branch first** — do not commit to main. Do not ask for confirmation; default to creating the branch.

```bash
git checkout -b <type>/<TICKET>-<short-description>
```

Branch naming: `<type>/<TICKET>-<short-kebab-description>`

Examples:
- `feat/RG-101-update-profile`
- `fix/CLIEN-45-sidebar-console-errors`
- `refactor/JAR-200-extract-validation`

### Step 3: Stage Changes

Run `git status` to see all changes. **Only stage files related to the current session's work.** Do not stage unrelated changes.

```bash
git status
```

- Review the list of modified/added/deleted files
- Stage only files that are part of the current task
- If unsure whether a file is related, ask the user
- Use `git add <specific-files>` — avoid `git add .` unless all changes are related

## Commit Message Format

```
<type>(<scope>): <subject>

<body>

Refs <TICKET>
```

The header is required. Scope is optional but recommended. Subject line must be under **50 characters**.

## Commit Types

| Type | Purpose |
|------|---------|
| `feat` | New feature |
| `fix` | Bug fix |
| `refactor` | Refactoring (no behavior change) |
| `perf` | Performance improvement |
| `docs` | Documentation only |
| `test` | Test additions or corrections |
| `chore` | Maintenance tasks |

## Common Scopes

| Scope | Package/Area |
|-------|-------------|
| `chat` | Chat UI and messaging |
| `session` | Session/auth handling |
| `iframe` | Iframe embeddable chat |
| `autopilot` | Autopilot/ITSM features |
| `itsm` | ITSM connections |
| `ui-schema` | Dynamic UI schema renderer |
| `db` | Database migrations |
| `api-server` | API server general |
| `client` | Client general |
| `storybook` | Storybook stories |
| `ci` | CI/CD workflows |
| `keycloak` | Keycloak theme/config |
| `k8s` | Kubernetes/deployment |

## Subject Line Rules

- Use imperative, present tense: "add feature" not "added feature"
- Lowercase first letter
- No period at the end
- Maximum **50 characters**

## Body Guidelines

- Explain **what** and **why**, not how
- Use imperative mood and present tense
- Include motivation for the change
- Contrast with previous behavior when relevant
- Optional — omit for trivial changes

## Footer: Ticket Reference

**Always** include the ticket reference in the footer:

```
Refs RG-101
```

- Always use `Refs` — never `Fixes` or `Closes`
- Use the ticket number provided by the user

## AI Attribution

**NEVER** add AI attribution to commits. No `Co-Authored-By`, no "Generated by AI", no Claude references anywhere in the commit message.

## Pre-commit Hooks

Husky runs automatically on `git commit`:
1. `npx lint-staged` → `biome check --write` + `biome lint`
2. `pnpm run type-check`
3. If api-server files staged: auto-generates OpenAPI spec

Do not manually run these checks before committing — the hooks handle it. If hooks fail, fix the issue and commit again.

## Examples

### Simple fix

```
fix(api): handle null response in user endpoint

User API could return null for deleted accounts, causing dashboard
crash. Add null check before accessing user properties.

Refs RG-234
```

### Feature with scope

```
feat(chat): add message reactions

Allow users to react to messages with emoji. Reactions display
below the message with a count badge.

Refs CLIEN-89
```

### Refactor

```
refactor(session): extract validation logic

Move duplicate validation from three endpoints into shared
validator. No behavior change.

Refs RG-150
```

### Documentation

```
docs(iframe): update integration guide

Refs CLIEN-42
```

### Multiple files, minimal body

```
chore(ci): update deploy workflow

Refs RG-300
```

## Principles

- Each commit should be a single, stable change
- Commits should be independently reviewable
- Only commit files related to the current task
- The repository should be in a working state after each commit
