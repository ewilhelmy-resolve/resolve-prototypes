# Kubernetes Deployment Guide

Deploying the Rita application to Kubernetes using Helm charts and helmfile orchestration.

## Architecture Overview

Rita is deployed as a multi-tenant microservices application on Kubernetes with:

- **API Server** - TypeScript/Express backend
- **Frontend** - React/TypeScript SPA with nginx
- **Centralized Keycloak** - Authentication (realm per tenant)
- **External Services** - PostgreSQL (RDS in prod), RabbitMQ, Consul, Valkey

Each tenant gets a dedicated Keycloak realm and K8s namespace.

### Environments

| Environment | Cluster | Keycloak | Actions API |
|-------------|---------|----------|-------------|
| Dev | Kops (`resolve-actions-k8s.local`) | `auth.dev.resolve.io` | `actions-api-dev.resolve.io` |
| Prod | EKS (`rita-prod`) | `auth.resolve.io` | `actions-api-prod.resolve.io` |

## Deployment Methods

### 1. Automated via GitHub Actions

#### Dev (`deploy-k8s-dev.yml`)

Triggers on:
- Push to any branch except `main`
- Manual workflow dispatch (with optional `namespace` and `services` inputs)

Namespace is auto-detected from the triggering user via `match_user_tenant_rita` workflow, or can be specified manually.

#### Prod (`deploy-k8s-prod.yml`)

Triggers on manual workflow dispatch only, with inputs:
- **target**: `single` or `all` (deploys one customer or all from `customers.json`)
- **customer**: Namespace name (required if target=single)
- **services**: `all`, `api-server`, or `frontend`
- **image_tag**: Skip build and use existing image tag

Prod deploys run sequentially per customer (`max-parallel: 1`).

### 2. Manual via Helmfile

#### Dev

```bash
cd helm/rita/ops

# Set required env vars
export IMAGE_TAG=<commit-sha>
export ECR_REGISTRY=432993365903.dkr.ecr.us-east-1.amazonaws.com
export DATABASE_URL=<connection-string>
export RABBITMQ_URL=<connection-string>
export CONSUL_HTTP_TOKEN=<token>
export VALKEY_URL=<connection-string>
export AUTOMATION_WEBHOOK_URL=<webhook-url>
export AUTOMATION_AUTH=<token>
export DEV_KEYCLOAK_ADMIN=admin-dev
export DEV_KEYCLOAK_ADMIN_PASS=<password>
export DEV_RITA_CLIENT_SECRET=<secret>

helmfile -e dev \
  --state-values-set namespace=<tenant> \
  apply
```

#### Prod

```bash
cd helm/rita/ops

export IMAGE_TAG=<commit-sha>
export ECR_REGISTRY=432993365903.dkr.ecr.us-east-1.amazonaws.com
export DATABASE_URL=<built-from-rds-admin-url>
export RABBITMQ_URL=<connection-string>
export CONSUL_HTTP_TOKEN=<token>
export VALKEY_URL=<connection-string>
export RDS_ADMIN_URL=<admin-connection-string>
export PROD_KEYCLOAK_ADMIN=<admin-user>
export PROD_KEYCLOAK_ADMIN_PASS=<password>
export PROD_RITA_CLIENT_SECRET=<secret>
export AWS_ACCESS_KEY_ID=<key>
export AWS_SECRET_ACCESS_KEY=<secret>

helmfile -e prod \
  --state-values-set namespace=<customer> \
  apply
```

**Prod DATABASE_URL** is built dynamically from `RDS_ADMIN_URL` by replacing the database name with the customer namespace. The workflow handles this automatically.

## Helm Hook Execution Order

Jobs run as pre-install/pre-upgrade hooks in this order:

| Weight | Job | Chart | Purpose |
|--------|-----|-------|---------|
| -20 | api-server-secret | api-server | Creates K8s secret with connection strings |
| -10 | db-init | api-server | Creates customer database if not exists |
| 5 | keycloak-client-import | frontend | Creates/updates Keycloak realm and clients |
| 5 | migration | api-server | Runs database migrations |
| 6 | automation-aws-secret | api-server | Creates K8s secret with AWS/Keycloak admin creds (prod only) |
| 7 | automation-creds | api-server | Creates RITA Event Key and stores in Secrets Manager (prod only) |
| 10 | route53-dns | frontend | Creates DNS CNAME record (post-install/post-upgrade) |

## Automation Credentials (Prod)

The automation-creds K8s job handles per-tenant RITA Event Key creation:

1. Checks AWS Secrets Manager for existing credentials
2. If not found:
   - Creates a temporary Keycloak admin user in the tenant realm
   - Assigns the `Administrator` role (client: `actions`)
   - Authenticates as temp user to get Actions API token
   - Retrieves `Events Client ID` and `Tenant ID`
   - Creates a `RITA Event Key` via the Actions API
   - Builds `AUTOMATION_AUTH` as `base64(events_client_id:secret_key)`
   - Stores both `automation-auth` and `automation-webhook-url` in Secrets Manager
   - Deletes the temporary user
3. After helmfile apply, the workflow patches the `api-server-secret` K8s secret with values from Secrets Manager

On subsequent deploys, the job skips entirely (credentials already exist).

## Keycloak Client Management

### Automated Client Import

Runs as a pre-install/pre-upgrade hook on the frontend chart for both dev and prod.

**Creates/updates:**
- `onboarding-auth-client` (public) — frontend authentication
- `rita-go-auth-client` (confidential) — created once with auto-generated secret, skipped on subsequent deploys

The realm is created automatically if it doesn't exist.

### Internal Network Access

Both Keycloak instances (`auth.dev.resolve.io`, `auth.resolve.io`) resolve to internal ALBs via Route53 private hosted zones. The following VPCs must be associated with the private hosted zones:

- **Dev private zone**: Kops VPC (`vpc-008991f2f1947f8b3`)
- **Prod private zone**: EKS VPC (`vpc-04cb863f39eaf0919`)

The prod EKS VPC requires VPC peering with the platform VPC (`vpc-0490a931bc8e39bae`) for traffic to reach the internal ALBs.

## DNS Configuration

### Automated (Route53 Job)

Route53 CNAME records are created/deleted automatically by the frontend chart's post-install/post-upgrade hook.

Pattern: `<namespace>-rita.resolve.io` → ALB DNS

Requires:
- `ROUTE53_ENABLED=true`
- `ROUTE53_HOSTED_ZONE_ID` set
- IRSA role for Route53 (prod) or node IAM role (dev)

### Manual

```bash
kubectl get ingress frontend -n <namespace> -o jsonpath='{.status.loadBalancer.ingress[0].hostname}'
```

Create a CNAME record: `<namespace>-rita.resolve.io` → ALB DNS

## Database

### Dev

Uses a shared PostgreSQL instance. `DATABASE_URL` is passed directly via GitHub secret.

### Prod

Uses RDS PostgreSQL with pgvector. Per-customer databases are created automatically by the db-init job.

`DATABASE_URL` is constructed from `RDS_ADMIN_URL` by replacing the database name with the customer namespace. Connection strings must include `?sslmode=no-verify` for RDS SSL.

## Rollback

```bash

helm history api-server -n <namespace>
helm rollback api-server <revision> -n <namespace>

# All services

  --state-values-set namespace=<namespace> \
  apply --set image.tag=<previous-sha>
```

## Troubleshooting

### Kubeconfig Expired (Prod EKS)

```bash
aws eks update-kubeconfig --name rita-prod --region us-east-1
```

### ResourceQuota Errors

All containers (including init containers and job containers) must specify resource requests and limits when `resourceQuota` is enabled.

### Docker Hub Rate Limiting

The build step clears Docker credentials before building to avoid ECR auth interfering with Docker Hub pulls. If you hit 429 errors, wait 5-10 minutes and retry.

### Keycloak Import Failures

Check if the pod can reach Keycloak internally:
```bash
kubectl run -it --rm dns-test --image=curlimages/curl --restart=Never -n <namespace> \
  --overrides='{"spec":{"containers":[{"name":"dns-test","image":"curlimages/curl","command":["sh","-c","curl -sf https://auth.resolve.io/realms/master | head -5"],"resources":{"requests":{"cpu":"50m","memory":"64Mi"},"limits":{"cpu":"100m","memory":"128Mi"}}}]}}' \
  -- sh
```

If it hangs, check VPC peering and private hosted zone associations.

### Automation Creds Job Failures

Common issues:
- **OOMKilled**: `amazon/aws-cli` image is heavy; memory limit is set to 512Mi
- **"Account is not fully set up"**: User creation must include `email`, `firstName`, `lastName`, `emailVerified: true`, and `requiredActions: []`, then explicitly clear required actions via PUT
- **WAF blocking**: Job must run inside the cluster (K8s job), not from GitHub runner

### Stuck Helm Releases

```bash
helm list -n <namespace> -a
helm rollback <release> 0 -n <namespace> --force
# or
helm uninstall <release> -n <namespace>
```
